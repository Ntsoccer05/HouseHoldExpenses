# 機能・API 追加時のチェックリスト

このチェックリストは、新しい機能やAPI を追加する際の確認項目です。

## フェーズ1: 壁打ち & アイデアドキュメント作成

### 要件確認
- [ ] **対象スコープが明確か**
  - 例: 「全ユーザー対象」「特定カテゴリのみ」「管理者のみ」
- [ ] **ユースケースが明確か**
  - 例: 「支出入力時に即座に検証」「日1回の定期実行」
- [ ] **データ保持期間は決まっているか**
  - 例: 「通知は7日間保持」「ログは30日保持」

### 設計確認
- [ ] **バックエンド側の変更が明記されているか**
  - 新テーブル？既存テーブルへの列追加？
  - 新 API エンドポイント？既存エンドポイント拡張？
  - 新 Job/Notification？

- [ ] **フロントエンド側の変更が明記されているか**
  - 新ページ？既存ページ改修？
  - 新コンポーネント？

- [ ] **DB スキーマが具体的か**
  - テーブル名、カラム名、型、制約を記載
  - 外部キーの関係を図示

- [ ] **API 仕様が具体的か**
  - エンドポイント、HTTPメソッド、リクエスト/レスポンス形式
  - ステータスコード、エラーハンドリング

### テスト計画確認
- [ ] **テストケースが具体的か**
  - 正常系、異常系、エッジケース
  - パフォーマンス要件（負荷テスト）
  - セキュリティ（入力値チェック、認可）

### アイデアファイル確認
- [ ] **docs/ideas/pending/feature-xxx.md が作成されているか**
- [ ] **テンプレートに従っているか** → `docs/ideas/README.md` 参照

---

## フェーズ2: 実装（/add-feature）

### 実装前確認
- [ ] **docs/ideas/pending/feature-xxx.md の内容を最終確認した**
- [ ] **不明な点・修正すべき点はないか**
  - あれば、実装開始前にユーザーと相談

### 実装中の確認
- [ ] **tasklist.md に従って実装しているか**
  - [ ] → [x] に更新し忘れはないか
  - タスク完了ごとに更新

- [ ] **Laravel コマンド（php artisan make:*）を使用しているか**
  - `make:model`, `make:controller`, `make:migration` など
  - 手動作成を避ける

- [ ] **既存パターンに従っているか**
  - 命名規則
  - ディレクトリ構成
  - コードスタイル（docs/development-guidelines.md 参照）

- [ ] **API エンドポイントが routes/api.php に登録されているか**

- [ ] **Filament リソースが作成されているか**（管理者機能が必要な場合）
  - CRUD インターフェース自動生成確認

- [ ] **フロントエンド ルートが app.tsx に登録されているか**

### 実装後確認
- [ ] **すべてのタスクが [x] になった**
- [ ] **tasklist.md に漏れたタスクがない**
  - 見落としがないか、最後に確認

- [ ] **API 仕様書（docs/ideas/pending/feature-xxx.md）と実装が一致しているか**
  - エンドポイント、リクエスト形式、レスポンス形式

- [ ] **DB マイグレーションが実行されたか**
  - php artisan migrate が正常に完了

- [ ] **テーブル・カラムが仕様通りか**
  - PhpMyAdmin で確認

- [ ] **Filament 管理画面が正常に動作するか**
  - `/admin` にアクセスして CRUD 操作確認

## フェーズ3: 品質確認・振り返り

### 品質確認
- [ ] **implementation-validator で検証済み**
  - スペック準拠確認
  - コード品質確認
  - セキュリティ確認

### 振り返り記録
- [ ] **tasklist.md に振り返りを記録**
  - 完了日
  - 計画 vs 実績の差分
  - 学んだこと
  - 次回への改善提案

### 完了処理
- [ ] **docs/ideas/done/ に移動**
  - `mv docs/ideas/pending/feature-xxx.md docs/ideas/done/`

- [ ] **永続ドキュメント更新が必要か判断**
  - [ ] `docs/product-requirements.md` - 新機能を追記？
  - [ ] `docs/functional-design.md` - データモデル・機能仕様を追記？
  - [ ] `docs/architecture.md` - アーキテクチャ変更を追記？
  - [ ] `docs/glossary.md` - 新規用語を追記？

## よくあるミス

❌ **やってはいけないこと**:

1. **壁打ちをスキップ**
   - 「思いついたらすぐ /add-feature」をしない
   - 必ず docs/ideas/pending/ に設計書を作成してから

2. **tasklist.md を更新しない**
   - 実装中に「あとでまとめて更新しよう」としない
   - タスク開始時に [ ] → [x] に更新

3. **既存パターンを無視**
   - 独自のコーディング規約を使わない
   - 既存ファイルと同じスタイルに合わせる

4. **テスト計画を後回しにする**
   - テストケースは実装前に定義すべき
   - 実装後の「思いつき」では漏れやすい

5. **API 仕様書を作らない**
   - フロントエンド開発者が困る
   - 実装前に明確な仕様を docs/ideas/ に記載

6. **振り返りを書かない**
   - 「とりあえず完了した」と tasklist.md に記録しない
   - 次のプロジェクト改善に役立つので、重要

## テンプレート

新しい機能を追加する際は、以下をコピーして `docs/ideas/pending/` に作成：

```markdown
# [機能名]

**タイプ**: Feature / Improvement / Bugfix  
**優先度**: High / Medium / Low  
**ステータス**: Pending  

## 概要
[簡潔に説明]

## 背景・動機
[なぜ必要か]

## 要件

### 機能要件
- [ ] 要件1
- [ ] 要件2

### 非機能要件
- [ ] 要件1

## 設計・実装方針

### アーキテクチャ
[バックエンド側の変更]
[フロントエンド側の変更]

### データモデル
[SQL例]

### API 仕様
[エンドポイント例]

## 影響範囲

**バックエンド**:
- [変更するファイル]

**フロントエンド**:
- [変更するファイル]

## テスト計画
- [ ] テストケース1
- [ ] テストケース2

## 実装順序
1. [ ] タスク1
2. [ ] タスク2
```
