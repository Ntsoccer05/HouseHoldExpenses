# Ideas Management - アイデア管理ガイド

このディレクトリは、機能提案・改善アイデア・バグフィックスアイデアを体系的に管理します。

## ワークフロー

```
1. 壁打ち（Claude Code との対話）
   ↓
   「こういう機能が欲しい」「ここを改善したい」という要望を明確化
   ↓
2. アイデアドキュメント作成
   ↓
   docs/ideas/pending/ に Markdown ファイルを作成
   設計・実装方針を詰めたドキュメント化
   ↓
3. /add-feature で実装開始
   ↓
   壁打ちで作成したドキュメントを参照しながら実装
   ↓
4. 実装完了後、ファイルを docs/ideas/done/ に移動
```

## 他のドキュメントとの関係

アイデア段階での設計は、プロジェクトの永続ドキュメントと連携します：

| ドキュメント | 役割 | 更新時期 |
|------------|------|---------|
| `../product-requirements.md` | プロジェクト全体のユーザー要件・成功指標 | アイデア内容がプロジェクト要件に該当する場合 |
| `../functional-design.md` | 機能仕様・データモデル | アイデアの設計方針を確定後 |
| `../architecture.md` | システム構成・責任分離 | アーキテクチャに影響がある場合 |
| `../development-guidelines.md` | コーディング規約・開発プロセス | アイデア実装時の規約確認 |
| `../glossary.md` | 用語集・定義 | 新規用語が登場した場合 |

**アイデア作成時の確認事項**:
- [ ] アイデアが既存の `development-guidelines.md` のコーディング規約に従っているか
- [ ] `architecture.md` で定義されたレイヤー構造を破壊していないか
- [ ] `glossary.md` で既に定義されている用語を使用しているか（新規用語は定義が必要）
- [ ] このアイデアが `product-requirements.md` の一部として記載すべきか

## ディレクトリ構造

```
docs/ideas/
├── pending/                    # 検討中・実装待ちのアイデア
│   ├── feature-*.md           # 新機能
│   ├── improvement-*.md       # 改善
│   └── bugfix-*.md            # バグフィックス
├── done/                       # 実装完了したアイデア
│   ├── feature-*.md
│   ├── improvement-*.md
│   └── bugfix-*.md
└── README.md                   # このファイル（ガイド）
```

## ファイル命名規則

**形式**: `{type}-{priority}-{description}.md`

**type**: 
- `feature-` : 新機能
- `improvement-` : 既存機能の改善・リファクタリング
- `bugfix-` : バグ修正

**priority** (オプション):
- `high-` : 優先度高
- `mid-` : 優先度中
- `low-` : 優先度低

**description**: 
- ケバブケース（小文字＋ハイフン）
- 20 文字以内が目安

**例**:
- `feature-budget-alert.md`
- `feature-high-daily-report.md`
- `improvement-expense-form-validation.md`
- `bugfix-calendar-timezone-bug.md`

## 設計・実装方針

### アーキテクチャ
壁打ちで詰めた設計方針：
- バックエンド側の変更（API、データモデル）
- フロントエンド側の変更（UI、状態管理）
- データベース変更（テーブル追加・カラム追加など）

### 技術スタック
使用する技術・ライブラリ

**例**:
```
バックエンド: 
- Laravel: 新コントローラー作成、API エンドポイント追加

フロントエンド:

- React Hook Form: 予算入力フォーム
```

## ベストプラクティス

### ✅ すべきこと

1. **明確なスコープ定義**
   - 「何を」「なぜ」を最初に明確にする
   - スコープクリープを避ける

2. **具体的な実装方針**
   - 抽象的な記述は避ける
   - コード例・SQL 例を含める
   - ファイルパスを明示する

3. **テスト計画を事前に考える**
   - 実装と同時にテストケースを洗い出す
   - エッジケースを意識する

4. **壁打ちで詰める**
   - アーキテクチャ判断は実装前に確定
   - 後から「変更が大きくなる」を避ける

5. **既存コードとの整合性**
   - 既存パターンを参照する
   - 新しいパターンは正当化する

### ❌ すべきでないこと

1. **曖昧な記述**
   ```markdown
   ❌ 「予算機能を追加する」
   ✅ 「ユーザーがカテゴリごとに月間予算を設定し、超過時にメール通知される」
   ```

2. **スコープの拡張**
   - 「ついでにこれも」は別アイデアにする
   - pending ファイルは「これだけ」にフォーカス

3. **実装詳細の欠落**
   ```markdown
   ❌ 「REST API を追加する」
   ✅ 「POST /api/budgets エンドポイントを追加。
        リクエスト: { user_id, category_id, limit_amount }
        レスポンス: { id, status: 201 }」
   ```

4. **テスト計画なし**
   - 実装前に「何をテストするか」を決める

5. **後付けの修正**
   - ファイルを作成した後で大幅修正しない
   - 修正が必要なら、/add-feature 実行前に対話で詰め直す

## 実装開始前のチェックリスト

ファイル作成完了後、以下を確認してから `/add-feature` を実行：

- [ ] スコープが明確か（曖昧な部分はないか）
- [ ] 設計・実装方針が具体的か（実装者が迷わないか）
- [ ] テスト計画が定義されているか
- [ ] ファイルパス・テーブル名・API エンドポイントが明確か
- [ ] 既存コードとの相違点が説明されているか
- [ ] 優先度・タイプが正確に記載されているか

## 実装完了後

1. ファイルを `done/` フォルダに移動
2. 「完了日」セクションを記入
3. 実装時の変更点があれば追記（参考資料として）

```markdown
## 実装完了情報

**完了日**: 2026-04-12  
**実装者**: Ntsoccer05  
**関連 PR**: #123  

### 変更点（実装後に記入）
- 予期しない変更があった場合の備考
- パフォーマンス上の工夫
- テストカバレッジ結果
```

## 例

### Feature の例

**ファイル**: `pending/feature-monthly-report-export.md`

```markdown
# 月間レポート PDF エクスポート機能

**タイプ**: Feature  
**優先度**: High  
**ステータス**: Pending  

## 概要
ユーザーが指定月の支出レポートを PDF でダウンロードできる機能

## 背景・動機
- ユーザーから「レポートをプリントアウトしたい」という要望
- 銀行対応・税申告時に紙の記録が必要なケース
- 他の家計簿アプリにはある機能

## 要件

### 機能要件
- [ ] 月を選択する UI
- [ ] PDF 生成エンドポイント
- [ ] ダウンロード機能
- [ ] レポート内容: 月別集計、カテゴリ別内訳、グラフ

### 非機能要件
- [ ] PDF 生成時間: 3 秒以内
- [ ] ファイルサイズ: 5MB 以内

## 設計・実装方針

### アーキテクチャ
**バックエンド**:
- `BudgetReportController@export` 新規作成
- PDF ライブラリ: `laravel/laravel-pdf` または `barryvdh/laravel-dompdf`

**フロントエンド**:
- `ExportButton` コンポーネント（カレンダーページに追加）
- API 呼び出し → ファイルダウンロード

### テンプレート例
PDF では以下を含める：
- タイトル（ユーザー名、対象月）
- 月別集計（収入、支出、残額）
- カテゴリ別内訳（テーブル）
- グラフ（支出トレンド）
- 生成日時

## 影響範囲

**バックエンド**:
- `src/app/Http/Controllers/BudgetReportController.php` (新規)
- `src/resources/views/pdf/report.blade.php` (新規)

**フロントエンド**:
- `src/resources/js/components/ExportButton.tsx` (新規)
- `src/resources/js/pages/CalendarPage.tsx` (修正)

## テスト計画
- [ ] PDF ファイルが正しく生成されるか
- [ ] ファイル名に月が含まれるか
- [ ] 複数月連続エクスポート時にファイルが重複しないか
- [ ] 大量データ（1000+ 件支出）でのパフォーマンス

## 実装順序
- [ ] PDF テンプレート（Blade）の設計
- [ ] バックエンド API 実装
- [ ] フロントエンド UI 実装
- [ ] テスト・デバッグ

## 参考資料
- `docs/functional-design.md` 「レポート機能」セクション
```

## まとめ

このワークフローで得られるメリット：

✅ **実装時の迷い減少** - ドキュメントがあれば実装に集中できる  
✅ **品質向上** - 事前設計でバグが減る  
✅ **歴史管理** - 「なぜこう実装した」を後から確認できる  
✅ **複数人対応** - 他者が実装する際に設計が明確  
✅ **意思決定記録** - 壁打ちで出た判断が残される
